# LOGOS_MEMORY_SPEC_v1 — Память Logos (Vector RAG + GraphRAG) + Provenance

**Статус:** 📋 DRAFT  
**Версия:** 1.0  
**Дата:** 2025-12-22  
**Цель MVP:** обеспечить базовую память Logos на основе данных из Gmail/IMAP и Telegram:
- быстрый “слепок” (fast snapshot),
- глубокая индексация (deep index),
- поиск/recall для ответов и предложений действий,
- provenance и экспорт/удаление.

---

## 1) Принципиальная модель

Логос хранит знания в 3 контурах:

1) **Short‑term** (оперативная): контекст текущего сеанса/диалога/задачи  
2) **Vector RAG** (семантическая): фрагменты текста + embeddings  
3) **Graph memory** (структурная): сущности и связи (люди/проекты/обязательства/события)

Нормативно: **любая запись памяти должна быть привязана к источнику (provenance)**.

---

## 2) Namespace и владение

### 2.1 Namespace
Все данные памяти разделены по владельцу/логосу:
- `namespace = did:logos:{HANDLE}` (MVP) или `logos.account` (строже).

### 2.2 Доступ “только владелец”
Любые операции, меняющие долговременную память (Vector/Graph), должны выполняться только по контексту владельца:
- подключение новых источников,
- запуск индексации,
- удаление/очистка памяти,
- экспорт.

В автономном режиме Logos может **добавлять** знания в свою память (из разрешённых источников), но политика источников и подключений изменяется только владельцем.

---

## 3) Provenance (источник истины)

Каждый фрагмент памяти должен иметь provenance:

```json
{
  "sourceType": "email|telegram|session|manual",
  "sourceId": "email:...|tg:...|session:...",
  "capturedAt": "iso",
  "capturedBy": "logos-agent|owner",
  "hash": "0x...",
  "redactions": ["emails", "phones"]
}
```

**Требования:**
- `sourceType/sourceId` MUST быть заполнены (кроме ручных заметок).
- `hash` SHOULD быть хешем исходного текста после нормализации/редакций.
- Если данные менялись (суммаризация, редактирование), это фиксируется как `derivedFrom`.

---

## 4) Vector RAG (семантическая память)

### 4.1 Единица хранения: Chunk

```json
{
  "id": "uuid",
  "namespace": "did:logos:GUVUIK",
  "text": "string",
  "embedding": "[float...]",
  "metadata": {
    "kind": "email|tg|profile|note",
    "title": "string|null",
    "tags": ["string"],
    "people": ["person:..."],
    "projects": ["project:..."]
  },
  "provenance": {}
}
```

### 4.2 Ingestion pipeline
1) fetch события/письма  
2) normalize (html→text, убрать подписи, квотинг, трекинг)  
3) chunking (по абзацам/предложениям + лимит символов)  
4) embeddings  
5) upsert chunks  

### 4.3 Recall API (концептуально)
- `recall(query, k, filters)` → top‑K chunks + provenance
- `summarize(chunks)` → краткий контекст с ссылками на provenance

---

## 5) Graph memory (GraphRAG)

### 5.1 Сущности (минимум MVP)
- `Person` (контакт/участник переписки)
- `Project` (название/контекст)
- `Topic` (тема)
- `Commitment` (обязательства/дедлайны)
- `Channel` (tg‑канал/чат)
- `EmailThread` (цепочка писем)

### 5.2 Связи
- `Person ↔ Person` (communicates_with)
- `Person → Project` (involved_in)
- `EmailThread → Project/Topic`
- `Channel → Topic`
- `Commitment → Person/Project`

### 5.3 Извлечение сущностей
- из email/tg вытаскиваем кандидатов: имена, организации, проекты, дедлайны, задачи
- каждое утверждение должно иметь provenance (какое письмо/сообщение)

---

## 6) Fast Snapshot vs Deep Index

### 6.1 Fast Snapshot (≤ 10 минут)
Цель: быстрый полезный эффект.
- берём последние N писем + M сообщений tg
- делаем лёгкую классификацию и summary
- строим базовые сущности: топ людей/тем/проектов

### 6.2 Deep Index (фон)
Цель: полнота и устойчивость.
- глубокая индексация выбранных источников
- обновление графа
- периодические сводки (“что изменилось”)

---

## 7) Экспорт/удаление

### 7.1 Экспорт
- экспортируем: chunks + граф + настройки/summary
- формат: архив JSONL + graph schema + provenance refs
- подпись: желательно подписать экспорт owner key (чтобы можно было доказать происхождение)

### 7.2 Удаление
- пользователь должен иметь возможность удалить:
  - конкретный источник (например, один чат tg),
  - период (последние 30 дней),
  - всю память.
- удаление фиксируется как событие (audit)

---

## 8) Связь с “Сеансом с Логосом”

Результаты интервью сохраняются как:
- vector chunks (предпочтения/цели/стиль)
- graph nodes (цели, проекты, люди) — если пользователь назвал

Важно: данные “субъективных предпочтений” помечаются как `sourceType=session` и не смешиваются с фактическими источниками (email/tg).

---

## 9) Acceptance Criteria

- [ ] Память разделена по namespace логоса.
- [ ] Любая запись имеет provenance.
- [ ] Fast Snapshot даёт: топ входящих тем/людей/проектов + summary.
- [ ] Recall возвращает контекст с ссылками на источники.
- [ ] Экспорт/удаление доступны владельцу.
