# Contributing to Logos (Spec Driven Development)

Спасибо за вклад в открытую экосистему Logos. Здесь описано, **как правильно предлагать изменения**, чтобы протокол оставался понятным, форкабельным и безопасным.

---

## Как устроен репозиторий

- `README.md` — точка входа для разработчиков
- `MVP_*` — спеки MVP (проверяемые требования + acceptance criteria)
- `ADR/` — Architectural Decision Records (почему приняли то или иное решение)
- `DEPLOYMENT.md` — публичные адреса деплоев и заметки по эксплуатации

---

## Процесс (Spec Driven Development)

Мы предпочитаем изменения через “сначала спецификация”:

1) **Опишите проблему/цель**
   - Issue в GitHub: что болит, зачем, кого затрагивает.

2) **Спека или ADR**
   - Если меняется поведение системы: обновите/добавьте `MVP_*.md`.
   - Если нужно принять архитектурное решение/компромисс: добавьте `ADR/ADR-xxx-*.md`.

3) **Acceptance Criteria**
   - В каждой спеки должны быть **проверяемые критерии** (что именно должно работать).

4) **Реализация**
   - После согласования спеки/ADR делаем код (в соответствующем репозитории: frontend, protocol, backend).

---

## Правила PR

- **Небольшие PR**: 1 тема = 1 PR.
- **Ссылки**: любые изменения должны ссылаться на пункт спеки/ADR.
- **Никаких секретов**: `.env`, приватные ключи, токены — нельзя в git.
- **Чёткие сообщения коммитов**: что изменили и зачем.

---

## Как форкнуть и отправить PR (GitHub)

1) Форкните репозиторий в GitHub.
2) Клонируйте форк:

```bash
git clone <your-fork-url>
cd logos
```

3) Создайте ветку:

```bash
git checkout -b docs/improve-readme
```

4) Внесите изменения, закоммитьте и запушьте:

```bash
git add -A
git commit -m "docs: improve developer onboarding"
git push origin docs/improve-readme
```

5) Откройте Pull Request в основной репозиторий.

---

## Как работать в Cursor (пример “как у нас”)

Рекомендуемый рабочий цикл:

1) Откройте папку репозитория в Cursor.
2) Начните с `README.md` и соответствующей спеки `MVP_*`.
3) Если требуется выбор/компромисс — создайте ADR в `ADR/`.
4) После фиксации спеки/ADR — делайте кодовые изменения (в нужном репозитории).
5) Перед PR убедитесь, что **не добавились секреты** (см. `SECURITY.md`).
